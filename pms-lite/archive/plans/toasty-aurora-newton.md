# 人员管理平台 · 轻量自研（单服务）起步方案

> 状态：待评审 ｜ 作者：WorkBuddy ｜ 日期：基于既有对话
> 关联产物：`人员管理平台_PRD.md`(v0.5)、`人员管理平台_研发拆解工作项.md`、`人员管理平台_切片SQL.sql`、`人员管理平台_起步清单.md`

## 1. 结论与工期

**推荐方案：轻量自研单服务**（FastAPI 后端 + 极简前端 + SQLite/Postgres），替代原先的 RuoYi-Vue-Plus 全栈。

| 路线 | 核心交付工期 | 适合你吗 |
|---|---|---|
| 轻量自研（推荐） | **核心 1.5–2 周**，全量约 3–4 周 | ✅ 单人可长期维护、逻辑可精确实现 |
| 零代码工具 | 2–5 天 | ⚠️ 历史单价追溯/预算告警受工具约束 |
| Excel+脚本 | 1–3 天 | ❌ 无 Web/无权限/无多端协同 |
| RuoYi 精简切片 | 2–3 周 | ⚠️ 仍是 Java+Vue 全栈，单人偏重 |

**核心工期拆解（按单人节奏，每天约 4–6 有效小时）：**

| 阶段 | 内容 | 工期 |
|---|---|---|
| D0 | Python 环境确认(已具备 3.11) + 项目骨架 + SQLite 初始化 | 0.5 天 |
| D1–D2 | 主数据：部门/员工/项目/日单价历史表 + CRUD API | 2 天 |
| D3 | 核算核心 `AccountingService`：服务天数、历史单价匹配、产值、项目总支出 + A/B/C 验证 | 1.5 天 |
| D4 | 项目成本录入明细 + 预算管控（超支标记/预警） | 1 天 |
| D5–D6 | 四角色权限（仅管理员可改单价）+ 极简前端页面 | 2 天 |
| D7 | 导入工具（CSV/Excel 导入员工、单价、服务记录）+ 本地验证 | 1 天 |

> 核心（主数据+核算+成本明细+权限+导入）≈ **1.5–2 周**。看板/报表/移动端接口列为后续批次，不影响首版可用。

## 2. 技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 后端 | **FastAPI** + SQLAlchemy 2 + Pydantic v2 | 异步、自动 OpenAPI 文档、校验强 |
| 数据库 | **SQLite**（起步）→ 可平滑换 Postgres | 单文件部署，零运维 |
| 权限 | **FastAPI 依赖注入 + 简单 RBAC**（4 角色） | 不用 Sa-Token，自己写一个 `get_current_user` + `require_role("admin")` |
| 前端 | **极简 SPA**：Vue3/React 单页 或 甚至 HTMX + 模板 | 先跑通功能，UI 从简；预留移动端接口（同一套 REST API） |
| 导入 | **FastExcel / pandas** 读 xlsx，批量校验后入库 | 复用 PRD 的导入工具需求 |
| 部署 | 单进程 `uvicorn` + 反向代理（或内网直接跑） | 出差/工地通过内网穿透或 VPN 访问 |

> 不引入 Redis、不引入工作流引擎、不引入多租户——单公司多部门足够。

## 3. 数据模型（复用切片 SQL，移植到 SQLite/SQLAlchemy）

直接复用 `人员管理平台_切片SQL.sql` 的表结构与种子数据，仅把方言换成 SQLite：

- `pms_emp_unit_price(emp_id, effective_date, price)`，`UNIQUE(emp_id, effective_date)` —— 历史单价追溯基础
- `pms_project(id, name, dept_id, leader_user_id, budget, status)`
- `pms_service_record(id, emp_id, project_id, start_date, end_date, days, matched_price, output_amount, price_missing)`
- 部门、员工、用户/角色表（轻量，4 角色：系统管理员/项目负责人/部门负责人/普通员工）

**历史单价匹配规则（核心，必须与 PRD v0.5 一致）：**
> 对每条服务记录，取 `effective_date ≤ 服务起始日` 中**最新**的一条日单价；若无则 `price_missing = true`，产值记 0 并标红。

## 4. 核心核算逻辑（AccountingService）

```
服务天数   = 截止日 − 起始日 + 1
匹配单价   = max(effective_date ≤ 起始日)   # 缺失则标记
个人产值   = 匹配单价 × 服务天数            # 缺失单价时产值=0
项目人工   = Σ(该项目下所有服务记录产值)
项目总支出 = 项目人工 + 项目其他成本(成本明细)
预算状态   = 项目总支出 > 预算 ? 超支 : 正常
```

验证用例（来自 PRD A/B/C 示例）：A=5000 / B=2400 / C=600；项目1 人工=5600；项目2=2400；A 的 3 月历史服务=2400（AC-13 历史追溯）。

## 8. 风险

| 风险 | 说明 | 缓解 |
|---|---|---|
| R1 前端耗时超预期 | 极简 UI 仍可能占时间 | 先 HTMX/模板，不追求组件化 |
| R2 权限遗漏 | 单价写接口未全部加 admin 守卫 | 单点封装 `require_role`，grep 复查 |
| R3 历史单价边界 | 跨年/同日多价/缺失 | D3 用 A/B/C + 边界用例覆盖 |
| R4 数据迁移 | 后续换 Postgres | 起步就用 SQLAlchemy，方言隔离 |
