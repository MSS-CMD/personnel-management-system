## 一、背景（Background）

公司电气工程团队经常出差，各成员常去不同项目现场，缺乏统一台账导致：公司无法实时掌握员工行程与在服务项目；员工产值、项目支出靠人工核算易错滞后；多人服务同一项目时成本分摊无系统核算。

本平台目标：录入员工日单价、行程、项目预算，并由系统按 `日单价 × 服务天数` 自动核算个人人工产值与项目人工支出；支持手工录入**项目成本明细**（差旅/材料/外包等）汇总为项目总支出；支持**历史单价追溯**（按服务日期匹配当时生效单价）；以看板呈现"员工→项目 / 个人产值 / 项目预算执行率"。

> 验收基准示例：
> A 员工：3 月单价 400，7 月涨到 500。7/1–7/10 在项目1（10天）→ 按 500 算 = 5000（人工）
> B 员工 单价400 × (7/5–7/10=6天) = 2400 → 项目2（人工）
> C 员工 单价200 × (7/8–7/10=3天) = 600 → 项目1（人工）
> 项目1 另录入差旅成本明细 800 → 项目1 总支出 = 人工(5000+600) + 800 = 6400；项目2 = 2400。

## 五、需求描述（EARS 原则）

### 5.1 组织与部门（FR-ORG）
- **[Ubiquitous]** 系统应始终在"单公司"框架下维护多个部门，每部门含唯一标识与名称。
- **[Ubiquitous]** 系统应始终将每位员工归属到一个部门，每个项目可归属到一个部门（允许"公司级/未分配"）。

### 5.2 员工与日单价（FR-EMP）
- **[Ubiquitous]** 系统应始终保存员工唯一标识、姓名、所属部门、岗位及**日单价历史（多条带生效日期的记录）**。
- **[Event-driven]** 当管理员新增/修改日单价时，系统应记录生效日期，作为该日期起的新单价。
- **[Unwanted]** 若同一员工同一生效日期存在两条单价，系统应拒绝保存并提示"该生效日期已存在单价"。
- **[State-driven]** 在"员工已停用"状态下，系统应禁止其关联新服务记录，历史产值保留可查。

### 5.3 项目与预算（FR-PRJ）
- **[Ubiquitous]** 系统应始终为每个项目保存标识、名称、归属部门、负责人与预算金额（默认0）。
- **[Event-driven]** 当管理员创建项目时，系统应将状态置"进行中"、执行率初始化 0%。
- **[Unwanted]** 若预算改为小于已发生总支出，系统应提示"预算低于已发生支出"并允许强制保存（标记超支）。
- **[State-driven]** 在"项目已结项"状态下，系统应冻结新服务记录与成本明细录入，历史可查。

### 5.4 服务记录（FR-SVC）
- **[Ubiquitous]** 系统应始终将一条服务记录关联一个员工与一个项目，含起止日期。
- **[Event-driven]** 当保存服务记录时，系统应自动计算服务天数 = 截止−起始+1（含首尾），并计入员工所属部门与项目所属部门。
- **[Unwanted]** 若起始晚于截止，系统应拒绝保存并提示"起始日期不能晚于截止日期"。
- **[Unwanted]** 若同员工同项目日期区间重叠，系统应提示"服务日期重叠"并阻止保存。

### 5.5 项目成本明细（FR-COST）
- **[Ubiquitous]** 系统应始终将一条成本明细关联一个项目，含成本类型、金额、发生日期、备注。
- **[Event-driven]** 当管理员保存成本明细时，系统应将其金额累加到所属项目"其他成本"，并重算项目总支出与执行率。
- **[Unwanted]** 若金额 ≤ 0 或成本类型为空，系统应拒绝保存并提示原因。
- **[Optional]** 若成本明细关联具体员工，系统应在看板中同时体现“该员工贡献的其他成本”。
- **[State-driven]** 在“项目负责人”角色下，系统应允许其对所负责的项目录入与修改成本明细（部门负责人/员工无此权限）。

### 5.6 核算引擎（FR-CAL）—— 主心骨
- **[Ubiquitous]** 系统应始终按 `个人人工产值 = 该记录服务日期所匹配生效日单价 × 服务天数` 计算每条服务记录（**历史追溯**）。
- **[Ubiquitous]** 系统应始终按 `项目人工支出 = Σ(该项目服务记录人工产值)`、`项目其他成本 = Σ(该项目成本明细金额)`、`项目总支出 = 项目人工支出 + 项目其他成本` 计算。
- **[Event-driven]** 当日单价/服务记录/成本明细/预算任一变更，系统应实时重算受影响产值、项目支出与执行率。
- **[Unwanted]** 若某服务日期无匹配生效单价（如早于任何单价记录），系统应将该记录标"单价缺失"置顶提示，不影响其他记录。
- **[State-driven]** 在"多员工同项目"状态下，系统应分别累计各员工人工后求和，并与逐条记录一致。

### 5.7 总览看板（FR-DASH）
- **[Ubiquitous]** 系统应始终为管理者提供看板：员工卡片（部门、在服务项目、人工产值）；项目卡片（预算 / 人工支出 / 其他成本 / 总支出 / 执行率）。
- **[Event-driven]** 当管理者打开看板，系统应展示最新实时数据（缓存刷新 ≤ 5 分钟）。
- **[Unwanted]** 若项目执行率 ≥ 100%，系统应红色高亮并置顶。
- **[Optional]** 若启用时间筛选，系统应按起止日期重算看板数值。

### 5.8 预算管控（FR-BUD）
- **[Event-driven]** 当项目执行率 ≥ 预警阈值（默认80%），系统应向管理者发出预警。
- **[Event-driven]** 当管理员调整预算，系统应记录调整前后值、操作人、时间。
- **[Unwanted]** 若预算调整致执行率跳变，系统应重新评估预警并刷新看板。

### 5.9 导入工具（FR-IMPORT）
- **[Ubiquitous]** 系统应始终提供员工、项目、服务记录、成本明细四类 Excel 导入模板。
- **[Event-driven]** 当管理员上传合规 Excel，系统应按模板逐行校验并写入；成功行入库，失败行汇总为"失败回执"供下载修正。
- **[Unwanted]** 若导入数据违反校验（如日期倒序、单价缺失），系统应跳过该行并记入回执，不中断整批。

### 5.10 权限（FR-AUTH）
- **[Ubiquitous]** 系统应始终区分“系统管理员 / 项目负责人 / 部门负责人 / 普通员工”四角色。
- **[State-driven]** 在"项目负责人"角色下（即被设为某项目"负责人"的用户），系统应仅展示其负责项目及服务于这些项目的员工的数据，并允许其录入/修改所负责项目的成本明细（部门负责人/员工无此权限）。**项目负责人不可修改员工日单价**（日单价修改仅系统管理员，见 5.2）。
- **[State-driven]** 在“部门负责人”角色下，系统应仅展示其管理部门内的员工、项目、服务记录、成本明细与产值。
- **[State-driven]** 在“普通员工”角色下，系统应仅展示本人服务记录与人工产值，隐藏其他员工及项目预算明细。
- **[Ubiquitous]** 系统应始终仅允许系统管理员进行员工/项目/服务记录的增删改与预算设置调整；项目负责人仅可查看预算与执行率，不可修改预算。

## 六、流程说明（Flows）

### 6.1 主流程
```
录入员工(部门+日单价历史) → 录入项目(部门+预算)
→ 录入服务记录(员工+项目+起止) ──按服务日期匹配单价×天数=人工产值
→ 录入成本明细(项目+类型+金额) ──累加其他成本
→ 聚合: 项目总支出=人工+其他 → 刷新看板/执行率
```

### 6.2 导入流程
```
下载模板 → 填充 → 上传 → 逐行校验
├─ 通过 → 入库
└─ 失败 → 生成失败回执(可下载修正重传)，已通过行保留
```

### 6.3 预算预警
```
总支出累计 → 执行率=总支出/预算
├─ ≥80% 黄警 ├─ ≥100% 红警置顶 └─ <80% 正常
```

## 七、交互说明（Interaction）

- **部门管理页**：部门列表，支持增改。
- **员工管理页**：表格含部门列；日单价显示历史（当前+历史生效记录）。
- **项目管理页**：含归属部门、负责人、预算；展示"人工/其他/总支出/执行率"进度条。
- **服务记录页**：表单含员工、项目、起止日期；保存即时显示天数、匹配单价与人工产值。
- **成本明细页**：表单含项目、成本类型、金额、日期、备注。
- **看板页**：员工卡片（部门、在服务项目、人工产值）；项目卡片（预算/人工/其他/总支出/执行率）；超支红字置顶。
- **导入页**：四类模板下载 + 上传 + 失败回执下载。

## 十、边界场景与异常状态（补充）

- **B-1 跨月/跨年**：服务天数连续计算不分段；成本明细按发生日期记录。
- **B-2 单价追溯**：产值按服务日期匹配当时生效单价；若该日期无匹配单价，标"单价缺失"并置顶，不计入直至补全。
- **B-3 员工停用/调部门**：停用后禁新建记录，历史保留；调部门后新记录计入新部门。
- **B-4 项目结项**：冻结录入，历史可查，执行率定格。
- **B-5 并发编辑**：后保存者基于最新版本（乐观锁/冲突提示）。
- **B-6 单价为0/空（无匹配）**：标"单价缺失"黄提示，不计入。
- **B-7 成本明细负额/零**：拒绝保存。
- **B-8 导入部分失败**：已通过行入库，失败行回执可重传。

## 十一、数据口径（已确认）

- 费用口径：人工产值 + 项目成本明细（差旅/材料/外包/其他）；**不含对客户开票**。
- 天数计算：含首尾两天；休息日默认计入（"休息日不计"为可选项，默认关）。
- **单价取值：历史单价追溯（按服务日期匹配当时生效单价）。**
- 预算口径：项目总预算单列，执行率 = 总支出 ÷ 预算；超支允许但红警。
- 部署：首期本地验证，架构预留可迁移公有云。
- 移动端：本版 PC；系统 API 优先设计，预留移动端接口。
- 组织：单公司、多部门（部门级数据隔离）。
- 导入：需 Excel 批量导入（员工/项目/服务记录/成本明细）。
- 单价修改权限：**仅系统管理员可修改员工日单价**；修改写入新的生效历史，不影响已发生服务记录的追溯核算（默认全局单价模型；若需"同一人不同项目不同单价"为后续增强）。项目负责人仅可对所负责项目录入/修改成本明细，不可修改员工日单价、预算或他人项目。
